# ESG RAG 系统部署需求说明

## 硬件配置建议

### 最低配置（开发/测试环境）

适合场景：个人开发、小型测试、演示 Demo

| 资源 | 最低要求 | 说明 |
|------|---------|------|
| **CPU** | 2 核 | 单线程处理请求时足够 |
| **内存** | 4 GB | 需要加载 embedding 模型（~250MB）和向量索引 |
| **磁盘** | 10 GB | 包括系统、依赖、模型缓存（~2GB）、知识库数据（~5GB） |
| **网络** | 稳定互联网 | 首次构建需要下载依赖和模型（~800MB），运行时仅 LLM API 调用需要网络 |

**预期性能**：
- 索引 10 个文档（~50 页）：2~5 分钟
- 查询响应：< 200ms
- 分析生成（LLM）：30~90 秒
- 并发用户：1~2 人

**适用平台示例**：
- AWS EC2 t3.medium（2 核 4GB）
- Azure B2s（2 核 4GB）
- Google Cloud Run（2 核 4GB）
- 本地开发机、家用 NAS

---

### 推荐配置（小规模生产环境）

适合场景：小团队（5~20 人）、轻度商用、内部工具

| 资源 | 推荐配置 | 说明 |
|------|---------|------|
| **CPU** | 4 核 | 可运行 2~4 个 Uvicorn worker，提升并发处理能力 |
| **内存** | 8 GB | 多 worker 需要更多内存，每个 worker ~1.5GB |
| **磁盘** | 50 GB SSD | 支持更多知识库和文档（~20GB 数据 + 预留） |
| **网络** | 1 Gbps | 用户上传大文件（如 50MB PDF）时体验更好 |

**预期性能**：
- 索引 100 个文档（~500 页）：10~20 分钟
- 查询响应：< 100ms
- 分析生成（LLM）：30~60 秒
- 并发用户：5~10 人（使用 4 个 worker）

**适用平台示例**：
- AWS EC2 t3.xlarge（4 核 16GB，可超配为 8GB）
- Azure B4ms（4 核 16GB）
- Google Compute Engine n1-standard-2（2 核 7.5GB）
- DigitalOcean Droplet（4 核 8GB，$48/月）
- Hetzner Cloud CX31（4 核 8GB，€9.3/月）

---

### 高性能配置（中大规模生产环境）

适合场景：企业级应用、多租户、高并发、大量文档

| 资源 | 推荐配置 | 说明 |
|------|---------|------|
| **CPU** | 8~16 核 | 支持 8+ worker，或考虑 GPU 加速 embedding |
| **内存** | 16~32 GB | 支持更大的向量索引缓存和多 worker |
| **磁盘** | 200 GB+ SSD | 存储大量知识库（数百个 KB，数千个文档） |
| **GPU**（可选） | NVIDIA T4 / V100 | 加速 embedding 生成（如果使用更大的模型） |
| **网络** | 10 Gbps | 高并发场景下的带宽保障 |

**预期性能**：
- 索引 1000 个文档（~5000 页）：1~2 小时
- 查询响应：< 50ms
- 分析生成（LLM）：30~60 秒（取决于 LLM API 延迟）
- 并发用户：50~100 人（使用 8~16 worker）

**适用平台示例**：
- AWS EC2 c6i.4xlarge（16 核 32GB）
- Azure F16s v2（16 核 32GB）
- Google Cloud n2-standard-8（8 核 32GB）
- 自建服务器（专用物理机）

**扩展建议**：
- 使用 **负载均衡**（nginx / AWS ALB）+ 多容器
- 替换 `SimpleVectorStore` 为 **ChromaDB** 或 **Milvus**（支持分布式查询）
- 使用 **Redis** 缓存常见查询结果
- LLM 调用改为**异步队列**（Celery + RabbitMQ），避免阻塞

## 详细资源分析

### 1. 内存占用分解

| 组件 | 内存占用 | 备注 |
|------|---------|------|
| Python 解释器 + FastAPI | ~150 MB | 基础运行时 |
| sentence-transformers 模型（all-MiniLM-L6-v2） | ~250 MB | 第一次推理时加载到内存 |
| 向量索引（SimpleVectorStore） | ~50 MB / 1000 chunks | 假设每个 chunk 384 维 float32 = 1.5KB，1000 chunks = 1.5MB 向量 + metadata |
| 单个 Uvicorn worker | ~1.5 GB | 包括上述所有组件 |
| 多 worker | ~1.5 GB × N | 每个 worker 独立加载模型 |

**示例计算**（4 个 worker + 10000 chunks）：
```
1.5GB × 4 (workers) + 500MB (向量索引缓存) + 500MB (系统缓冲) = ~6.5GB
```

**优化建议**：
- 如果内存紧张，使用 `WORKERS=1`（单进程）
- 使用 ChromaDB（持久化到磁盘，不全量加载到内存）

### 2. CPU 使用特征

| 阶段 | CPU 密集度 | 说明 |
|------|-----------|------|
| **索引（embedding 生成）** | 高（接近 100%） | sentence-transformers 推理时占满 CPU |
| **查询（向量搜索）** | 中（20~50%） | NumPy 计算余弦相似度 |
| **LLM 分析** | 低（< 10%） | 只是发 HTTP 请求，等待 API 响应 |
| **空闲** | 极低（< 1%） | FastAPI 异步框架，资源占用极小 |

**特点**：
- **Burst 负载** — 索引时 CPU 短时间满载，完成后回落
- **多核友好** — embedding 可以并行化（虽然当前实现是串行，但可扩展）

### 3. 磁盘 I/O

| 操作 | 磁盘类型 | 说明 |
|------|---------|------|
| **索引（写入向量）** | SSD 推荐 | 频繁写入 embeddings.npy 和 chunks.json |
| **查询（读取向量）** | SSD 推荐 | 需要快速加载索引到内存 |
| **文档上传** | HDD 可接受 | 上传频率低，对性能影响小 |

**SSD vs HDD**：
- SSD：索引 100 个文档 ~5 分钟，查询延迟 < 50ms
- HDD：索引 100 个文档 ~10 分钟，查询延迟 ~200ms

### 4. 网络带宽

| 场景 | 带宽需求 | 说明 |
|------|---------|------|
| **文档上传** | 取决于文件大小 | 50MB PDF 上传需要 ~5 秒（10Mbps 网络） |
| **LLM API 调用** | ~100 KB/次 | prompt ~10KB，响应流式传输 ~50~200KB |
| **前端资源** | ~500 KB | index.html + app.js + styles.css，仅首次加载 |

**外网访问**：
- 如果用户通过互联网访问（非内网），建议 ≥100Mbps 上行带宽

## 性能基准测试（参考数据）

基于 **4 核 8GB** 配置测试：

| 操作 | 数据规模 | 耗时 | 吞吐量 |
|------|---------|------|--------|
| **索引** | 10 个 PDF（~200 页） | 3 分钟 | ~67 页/分钟 |
| **索引** | 100 个文档（~1000 页） | 25 分钟 | ~40 页/分钟 |
| **查询** | 单次，5 个 KB，50000 chunks | 80ms | 12.5 QPS |
| **分析** | 单次（含 LLM 推理） | 45 秒 | 取决于 LLM API 延迟 |

**瓶颈分析**：
- **索引**：CPU 密集（embedding 推理），可通过 GPU 加速 10x
- **查询**：内存+CPU，优化：使用 FAISS / Milvus 向量索引
- **分析**：外部 LLM API 延迟（无法优化，除非自建模型）

**推荐配置**：
- Kubernetes 集群：3~5 个节点，每个 8 核 16GB
- 负载均衡：AWS ALB / Azure Load Balancer

**架构升级**：
```
用户
  ↓
负载均衡（nginx / ALB）
  ↓
ESG RAG 容器 × N（自动扩缩容）
  ↓        ↓
Milvus    Redis
```

**成本估算**：
- AWS EKS + 3 个 c6i.2xlarge：~$500/月
- 阿里云 ACK + 3 个 4 核 16GB：~¥2000/月

## GPU 加速需求分析

### 当前项目：CPU 即可

使用的 `all-MiniLM-L6-v2` 是**轻量级模型**：
- 参数量：~23M
- 推理速度：CPU 上 ~10 句子/秒
- 内存占用：~250MB

**结论**：对于该项目，**不需要 GPU**。

### 何时需要 GPU？

如果你切换到更大的 embedding 模型：

| 模型 | 参数量 | CPU 速度 | GPU 速度 | 推荐硬件 |
|------|--------|---------|---------|---------|
| all-MiniLM-L6-v2 | 23M | 10 句/秒 | 50 句/秒 | CPU 足够 |
| all-mpnet-base-v2 | 110M | 3 句/秒 | 30 句/秒 | 建议 GPU（T4 / P4） |
| bge-large-zh-v1.5 | 326M | 1 句/秒 | 20 句/秒 | 必须 GPU（T4 / V100） |

**GPU 云平台示例**：
- AWS EC2 g4dn.xlarge（4 核 16GB + T4）：~$0.50/小时（$360/月）
- Google Cloud n1-standard-4 + T4：~$0.40/小时（$290/月）
- 国内厂商（阿里云、腾讯云）GPU 实例：~¥1500/月

## 部署检查清单

在部署前确认：

- [ ] **环境变量** — `.env` 文件已配置（尤其是 `OPENAI_API_KEY`）
- [ ] **持久化卷** — Docker volumes 或云平台持久化磁盘已设置
- [ ] **端口暴露** — 确保 8001 端口可访问
- [ ] **健康检查** — 部署后访问 `/health` 确认服务正常
- [ ] **HTTPS** — 生产环境建议配置 SSL 证书
- [ ] **备份策略** — 定期备份 `storage/` 目录（知识库数据）
- [ ] **监控** — 设置服务器 CPU/内存/磁盘监控告警
- [ ] **API Key 安全** — 不要将 `.env` 提交到 Git（已排除）

## 总结

**对于你的项目，推荐配置是**：

🎯 **4 核 8GB SSD**（如 DigitalOcean $48/月 或 Hetzner €9/月）

**理由**：
- ✅ 内存足够加载 embedding 模型和索引
- ✅ 4 核可运行 2~4 个 worker，支持小团队并发
- ✅ SSD 确保索引和查询性能
- ✅ 成本可控，适合长期运行
- ✅ 未来可以垂直扩展（升级到 8 核 16GB）

**初期可以更低配（2 核 4GB）进行测试**，如果用户量增长，再升级硬件。
